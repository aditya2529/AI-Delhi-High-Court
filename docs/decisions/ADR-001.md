# ADR-001: Backend framework — FastAPI over Spring Boot

**Status:** Accepted (MVP) · **Date:** 2026-05-17 · **Deciders:** Arnav (proposer), human architect (approver)
**Supersedes:** none · **Superseded by:** none

---

## Context

The Delhi HC Case Tracker backend must:

1. Issue **async outbound HTTP** to the court site and hold dozens of concurrent sessions while users type CAPTCHAs (I/O-bound, long-lived idle).
2. Parse HTML server-side with mature scraping libraries.
3. Expose a small, contract-driven JSON API to a Next.js frontend.
4. Be runnable on a single small VM for the MVP, with a clear path to horizontal scaling.
5. Be operated by a small team (Arjun on backend, no dedicated DevOps).

The realistic finalists are **FastAPI (Python 3.11)** and **Spring Boot (Java 21)**. Node/Express was considered briefly but rejected on parsing-ecosystem grounds (BeautifulSoup is meaningfully better than Cheerio for the layered-fallback strategy described in `STRATEGIES.md §3`).

---

## Options Considered

| Option | Pros | Cons | Cost |
|---|---|---|---|
| **A: FastAPI + Python 3.11** | Native `asyncio`/`httpx`; best-in-class HTML parsing (BS4 + lxml); pydantic v2 auto-generates the OpenAPI spec from models (free contract); fast iteration; small team can ship; uvicorn is production-grade enough for MVP single-node | GIL caps CPU-bound scaling per process (mitigated by multi-worker + async I/O dominance here); typing discipline weaker than statically-compiled languages; packaging story (poetry / uv) requires conscious choice | Low dev cost; low ops cost; ~$15–25/mo VM |
| **B: Spring Boot + Java 21 (virtual threads)** | Statically typed; mature observability ecosystem; virtual threads handle the I/O-wait pattern well; enterprise-standard if the org has Java expertise | Heavier runtime (200–500MB JVM baseline vs ~80MB Python); HTML parsing (Jsoup) is decent but the layered CSS/XPath/regex fallback pattern is more idiomatic in Python; longer build/deploy cycle; team would need to ramp on Spring conventions; over-engineered for a 6-endpoint MVP | Higher dev cost (1–2 sprint ramp for Python-leaning team); 2–3x VM size; longer cold starts |
| **C: Node.js + Express + TypeScript** | Frontend team already knows TS; smaller cognitive switch | Cheerio parsing is weaker for our layered-fallback needs; no equivalent to lxml's robust HTML5 handling on broken markup; async model fine but no parsing ergonomic win | Comparable to A on infra; loses on parsing |

---

## Decision

**Option A — FastAPI on Python 3.11+.**

**Reasoning:**
1. **Workload fit.** The hot path is `await httpx.get(...)` then `BeautifulSoup(...)`. FastAPI + `httpx.AsyncClient` is the most direct expression of this. Spring Boot with virtual threads gets us to the same place with more ceremony.
2. **Parsing ecosystem.** Our parsing strategy depends on layered CSS → XPath → regex fallbacks (`STRATEGIES.md §3`). BS4 + lxml is the most mature stack for this; switching to Jsoup or Cheerio gives up real defensive capability.
3. **Auto-generated contract.** FastAPI emits OpenAPI from pydantic models — the API contract in `docs/api/API-CONTRACT.md` becomes executable. Spring Boot needs springdoc-openapi annotations, an extra ceremony.
4. **Team velocity.** A two-developer MVP ships faster in FastAPI. Spring's payoff (typed enterprise patterns) accrues over 12+ months and many endpoints — we have 6.
5. **Operational footprint.** Single VM, single Python process, two uvicorn workers is enough for ~60 sessions/min sustained, which exceeds MVP target.

---

## Consequences

**Gains (✅)**
- Fast iteration; tight loop between contract and implementation via pydantic.
- Async I/O matches the natural shape of the workload.
- Smallest credible production stack for the requirements.
- Easy onboarding for Python-fluent contributors.

**Trade-offs (⚠️)**
- Python typing is opt-in; we must enforce `mypy --strict` in CI to compensate (added as a CI gate from day 1).
- The GIL caps single-process CPU throughput. Mitigated because our workload is I/O-bound (~95% of wall time is waiting on the court site or the user). If parsing becomes a hot path (unlikely), we move it to a process pool.
- Hot reloads on uvicorn are dev-only; production runs without `--reload`.

**Lock-in (🔒)**
- **Low.** The architecture's boundaries (`SessionStore` protocol, `CourtClient` interface, `Parser` module) are framework-agnostic. A future port to Spring Boot or Go is plausible if the org standardizes elsewhere — the contracts in `docs/api/API-CONTRACT.md` are the durable artifact.
- **Migration cost estimate (worst case):** ~3 engineer-weeks to port to Spring Boot if mandated, dominated by re-implementing the parser and re-doing integration tests.

## Open Questions (for human architect)

1. Is there an existing org-standard backend stack we should respect? If "Java everywhere" is a real constraint, revisit this ADR.
2. Do we need to lock down a Python version above 3.11 (e.g., 3.12 for performance improvements)? Default: 3.11 for ecosystem maturity in 2026; revisit if uvicorn / pydantic v2 perf gains in 3.12 are material.
3. Package manager: `uv` (fast, modern) vs `poetry` (stable, established)? Default: `uv` for MVP speed; reversible.
