# ADR-003: Human-in-the-loop CAPTCHA solving — no model-based or third-party solving

**Status:** Accepted (MVP and beyond) · **Date:** 2026-05-17 · **Deciders:** Arnav (proposer), Sneha (security/legal), human architect (approver)
**Supersedes:** none · **Superseded by:** none

---

## Context

The Delhi High Court's public case-status form is protected by an image CAPTCHA. The product flow requires the user's case query to traverse this CAPTCHA before the result HTML can be parsed.

The technical options for crossing a CAPTCHA fall on a continuum:

1. **Human solves it** (the user types what they see).
2. **Local OCR / vision model solves it** (Tesseract, custom CNN, off-the-shelf foundation model with vision).
3. **Third-party CAPTCHA-solving service** (2captcha, anti-captcha, deathbycaptcha).
4. **Avoid the CAPTCHA entirely** (find a non-CAPTCHA-protected upstream endpoint, e.g. an API).

This decision is asked early because it is **architecturally load-bearing** — it shapes the session model (sliding human-time TTL), the API contract (CAPTCHA round-trip endpoints), the latency profile (15–60s human wait), and the legal posture of the product.

---

## Options Considered

| Option | Pros | Cons | Cost | Legal / ethical posture |
|---|---|---|---|---|
| **A: User solves CAPTCHA (current proposal)** | Respects the explicit anti-bot intent of the upstream operator; no third-party dependency; zero arms race with court IT; defensible posture; no cost per request | Latency includes human-typing time (15–60s); UX friction; user must be physically present | $0 per request | Aligned with the upstream operator's expressed intent. The user is solving the CAPTCHA the court designed for them; we are merely a UX layer. |
| **B: Local OCR / vision model** | Removes human-typing latency; "smoother" UX | Defeats the explicit anti-automation control. Court IT can (and likely will) respond with harder CAPTCHAs / IP blocks / legal action. Arms race we will lose. Even if technically possible, **demonstrates intent to bypass a security control** — bad legal posture. Maintenance cost: each CAPTCHA-style change requires retraining | Engineering: 2–4 weeks initial; ongoing maintenance; risk-weighted cost of legal exposure is **very high** | **Adversarial to the upstream operator.** Under Indian IT Act, Section 43(a) and 66, unauthorized circumvention of access controls of a computer resource is actionable. Court records are a government computer resource. Even if the underlying data is public, defeating the access control is the issue. |
| **C: Third-party CAPTCHA-solving service** | Same UX benefit as B; outsourced to vendor | All of B's legal issues, plus: leaks the user's case query to an external party; vendor T&Cs typically prohibit use against government sites; data residency concerns; per-request cost ($1–3 per 1000 CAPTCHAs); creates a permanent third-party dependency in our critical path | $1–3 / 1000 + vendor risk + legal | Same posture as B, plus a third-party data-leakage problem. **Worst of both worlds.** |
| **D: Find a non-CAPTCHA upstream endpoint** | Cleanest if available | We don't know if one exists. Even if it does, using a "hidden" endpoint that the operator deliberately did not surface raises the same circumvention question | Unknown until spike | Indeterminate — requires research. |

---

## Decision

**Option A — the user solves the CAPTCHA. Always. No exceptions in MVP, Phase 1, or Phase 2 without a new ADR superseding this one.**

**Reasoning:**

1. **Respects upstream intent.** The Delhi High Court placed a CAPTCHA on this page deliberately. A CAPTCHA is a Reverse Turing Test whose explicit purpose is "only humans should programmatically reach this endpoint at scale." Honoring that intent is not optional — it is the line between a "workflow simplification tool" (us) and an "automated scraper" (which we explicitly are not).
2. **Legal defensibility.** The product can be honestly described as "a better UI around a public form." Every CAPTCHA is solved by the human end-user. There is no automation of the access control. Under Indian IT Act and equivalent international jurisprudence, this is the defensible position.
3. **No arms race.** Option B starts an arms race with court IT — they will harden CAPTCHAs, add behavioral detection, rotate templates — and we will lose because they have institutional patience and we have a small team.
4. **Architectural simplicity.** The human-typing wait time (15–60s p95) is the dominant latency component, but it is *only* during active flows. The async backend handles dozens of waiting sessions cheaply. UX friction is real but manageable with a clear countdown timer and proactive refresh.
5. **No third-party risk.** Zero dependencies, zero per-request cost, zero data leakage.

---

## Consequences

**Gains (✅)**
- Clean legal/ethical posture; the product is honestly what it claims to be.
- No third-party in the critical path; reliability is bounded by us + the court site only.
- Zero per-request external cost.
- Architecturally simple: no model serving infra, no CAPTCHA-solving worker, no retraining pipeline.

**Trade-offs (⚠️)**
- **Latency.** Every flow incurs 15–60s of human time. Cannot be optimized away — only smoothed (good UI, countdown, proactive refresh).
- **Cannot be used in automation contexts.** A future product use-case like "scheduled daily case-status snapshots for a litigant" is **out of scope under this ADR** unless we explicitly negotiate a partner channel with court IT (formal MoU / API). This is a deliberate constraint.
- **UX cost.** Users who fat-finger the CAPTCHA add friction. Mitigated by 3-attempt cap with image refresh.

**Lock-in (🔒)**
- **Architectural lock-in is low** — the API and session design are fine for either choice.
- **Product / business lock-in is medium** — automation use cases are off the table. Reversing this decision would require: a new ADR, sign-off from legal, and almost certainly a formal arrangement with the Delhi High Court IT department. This is the right level of friction; the decision should be hard to reverse.

**Security inline note (Sneha):**
- We log nothing about the CAPTCHA image bytes or typed answers beyond an opaque counter.
- We do not store CAPTCHA images at rest beyond the session TTL (10 min max).
- A `2captcha.com` or similar domain appearing in our outbound DNS logs would be an immediate compliance violation — add to network egress denylist.
