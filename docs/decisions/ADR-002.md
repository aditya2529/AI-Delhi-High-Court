## Context

The MVP needs persistent storage for three things:

1. **`parsed_case` cache** â€” successful parses, keyed by `case_id`, TTL 6 hours. Read-heavy on cache hits, write-on-miss.
2. **`failure_log`** â€” recent failed requests for ops/admin observability, retention 30 days.
3. **Session store (optional fallback)** â€” if Redis is not deployed, the abstract `SessionStore` falls back to a SQLite-backed key-value table.

Constraints:
- Single VM in MVP; minimal ops overhead.
- Estimated MVP load: 60 concurrent sessions sustained, ~5â€“10 writes/sec peak (well within SQLite's documented capacity in WAL mode).
- Must be **easy** for a successor team to migrate to PostgreSQL when the system needs to scale horizontally (Phase 1).
- No team member is a full-time DBA; whatever we pick must be self-administering.

## Decision

**Option A â€” SQLite in WAL mode, accessed through SQLAlchemy 2.0 (async) with PostgreSQL-compatible types only, Alembic migrations from day 1.**

**Reasoning:**
1. **Right-sized for MVP.** ~10 writes/sec is well within SQLite WAL's comfort zone. WAL mode allows many concurrent readers while a single writer commits.
2. **Migration tax is paid upfront.** By using SQLAlchemy 2.0 with the **PostgreSQL dialect's type subset** (no `BIGSERIAL`, no `JSONB` â€” we use `String`/`JSON` portable types; no functional indexes that SQLite can't express) and Alembic from day 1, the day we flip the connection string to `postgresql+asyncpg://...`, the schema works. The MVP becomes a stepping stone, not a throwaway.
3. **Zero operational overhead.** No DB process to monitor, no backups except `cp sessions.db backup/`. Critical for a small team.
4. **Reversible.** Switching to Postgres is a config change + `alembic upgrade head` on the Postgres side. Estimated migration effort: 0.5 sprint.

**Constraints enforced from day 1:**
- All schemas live in Alembic migrations; no inline `CREATE TABLE`.
- SQLAlchemy column types restricted to the **intersection** of SQLite and PostgreSQL: `Integer`, `String`, `Text`, `DateTime`, `Boolean`, `JSON` (not `JSONB`).
- No SQLite-specific syntax (`INSERT OR REPLACE`, `AUTOINCREMENT`, `PRAGMA` outside of WAL setup). Use SQLAlchemy's `on_conflict_do_update` portable construct.
- `PRAGMA journal_mode=WAL` and `PRAGMA synchronous=NORMAL` set at connect time.
- `failure_log.created_at` indexed (Rohit: confirm).
- `parsed_case` primary key on `sha1(case_id)`.
- Foreign keys explicitly enabled (`PRAGMA foreign_keys=ON`).

## Consequences

**Gains (âś…)**
- MVP ships with zero DB ops burden.
- Tests run against the same engine as dev â€” fast, deterministic.
- Migration path is real, not aspirational â€” Alembic migrations are reused verbatim on Postgres.
- Backups are file copies.

**Trade-offs (âš ď¸Ź)**
- We give up Postgres-only features (`JSONB` query operators, full-text search, partial indexes on expressions). If any of these become essential before Phase 1, we migrate early â€” acceptable.
- Single-writer ceiling means we **cannot** scale beyond one application node while still on SQLite. This is by design; Phase 1 migration is the answer, not "hack around SQLite."
- WAL files must be on a real filesystem; **not** network-mounted (NFS breaks SQLite locking). Documented in runbook.

**Lock-in (đź”’)**
- **Very low.** The type-portable schema, the SQLAlchemy abstraction, and Alembic together make this a one-day migration when the time comes. The 2-week rewrite scenario (Option C) is avoided.
- Risk: a junior developer adds a SQLite-only construct (`INSERT OR IGNORE`, etc.) and breaks portability. Mitigation: CI runs migrations against an ephemeral Postgres container weekly to catch drift.
